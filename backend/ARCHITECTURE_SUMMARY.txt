================================================================================
                          THE 4-LAYER ARCHITECTURE
================================================================================

Layer 1: ROUTERS (HTTP Protocol)
├─ File: app/routers/*.py
├─ Responsibility: Parse requests, validate schemas, format responses
├─ No business logic, no database queries
└─ Example: @router.post("/projects") - just HTTP handling

Layer 2: SERVICES (Business Logic)
├─ File: app/services/*_service.py
├─ Responsibility: Rules, validation, workflows
├─ Calls repositories, never touches HTTP or database directly
└─ Example: ProjectService.create() - checks if name is valid

Layer 3: REPOSITORIES (Data Access)
├─ File: app/repositories/*_repo.py
├─ Responsibility: Database queries, pagination, optimization
├─ Only SQLAlchemy code goes here
└─ Example: ProjectRepository.list_by_org() - SELECT * FROM projects

Layer 4: MODELS (Data Structure)
├─ File: app/models/*.py
├─ Responsibility: Table definitions, relationships
├─ SQLAlchemy ORM definitions
└─ Example: class Project(Base) with columns and relationships

FLOW: Request → Router → Service → Repository → Database → Response

================================================================================
                          CURRENT STATE PROBLEMS
================================================================================

PROBLEM 1: Business Logic Scattered
├─ Logic in routers/auth.py (register, login)
├─ More logic in routers/analyses.py (validation, workflow)
├─ Yet more logic in tasks.py (analysis stages)
├─ Hard to test, hard to reuse
└─ SOLUTION: All logic in services/

PROBLEM 2: Tight Coupling
├─ Routers use SQLAlchemy directly
├─ SQL queries embedded in endpoints
├─ Hard to change database without rewriting routes
└─ SOLUTION: Repositories abstract database

PROBLEM 3: Multiple Model Files
├─ models.py (core models)
├─ models_phase1.py (phase 1 models??)
├─ models_phase2.py through models_phase5.py
├─ Confusing, incomplete refactoring
└─ SOLUTION: Merge into single models/ directory

PROBLEM 4: Hard to Test
├─ Can't test business logic without HTTP
├─ Can't test without real database
├─ Celery tasks duplicate logic
└─ SOLUTION: Services can be tested with mock repos

PROBLEM 5: Only 1 Service File
├─ services/ has only media_processor.py
├─ No consistent pattern for business logic
├─ New developers don't know where to put code
└─ SOLUTION: Expand services/ for all features
